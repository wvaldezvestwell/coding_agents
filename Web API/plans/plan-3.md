# Implementation Plan — 3: Configure JWT bearer authentication and protect API endpoints
> Auto-generated by /dev-start. Review before execution.

## Summary

The Web API (ASP.NET Core minimal API, .NET 10) currently serves every endpoint anonymously. This ticket adds JWT bearer-token validation to the request pipeline and denies anonymous access to `/weatherforecast` (and, by default, future endpoints). Token issuance is intentionally out of scope (ticket #4) — during development, tokens come from the `dotnet user-jwts` tool, which makes this slice independently testable. This is the first child of epic #2 (Add JWT authentication and authorization to the Web API).

## Scope

**In scope**
- Add the `Microsoft.AspNetCore.Authentication.JwtBearer` NuGet package
- Register authentication (JWT bearer scheme) and authorization services in `Program.cs`
- Add a `Jwt` configuration section (`Issuer`, `Audience`); signing key supplied via user-secrets / environment variables, never committed
- Add `app.UseAuthentication()` and `app.UseAuthorization()` to the pipeline
- Require authorization on `/weatherforecast` via `.RequireAuthorization()`
- Validate issuer, audience, lifetime, and signing key (HS256 symmetric key)

**Out of scope**
- Token issuance endpoint (`POST /auth/token`) — ticket #4
- OpenAPI bearer security scheme and integration test project — ticket #5
- Roles, policies, refresh tokens, user store

## Acceptance Criteria

- [ ] `GET /weatherforecast` without a token returns **401**
- [ ] `GET /weatherforecast` with a valid token (e.g. `dotnet user-jwts create`) returns **200**
- [ ] A token with wrong issuer, wrong audience, invalid signature, or expired lifetime returns **401**
- [ ] The signing key is not committed to the repo (user-secrets / env var only)
- [ ] (Implied) The 401 response carries the `WWW-Authenticate: Bearer` challenge header
- [ ] (Implied) `dotnet build` succeeds with no new warnings

## Testing Strategy

**Manual (Development environment):**
1. `dotnet run` from `Web API/`
2. `curl -i https://localhost:<port>/weatherforecast` → expect `401` with `WWW-Authenticate: Bearer`
3. `dotnet user-jwts create` (uses the project's issuer/audience config), then `curl -i -H "Authorization: Bearer <token>"` → expect `200` with forecast JSON
4. Tamper with the token payload / use an expired token → expect `401`

**Automated:** none in this ticket — the xUnit integration test project is ticket #5's deliverable. Build verification via `dotnet build`.

**Environments:** local Development only; no deployed environments exist for this project.
